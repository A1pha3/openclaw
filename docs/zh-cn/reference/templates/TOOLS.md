## 第一部分：核心概念与设计原理

### 1.1 为什么需要 TOOLS.md？

在深入了解 TOOLS.md 的具体用法之前，我们需要首先理解**设计者为什么引入这个单独的配置文件**。

**问题背景：配置信息的碎片化困境**

在复杂的计算环境中，用户和 AI 助手需要记住大量的配置信息：

| 配置类型 | 示例 | 存储位置 |
|----------|------|----------|
| 设备信息 | 相机名称、位置、访问地址 | 分散在各处 |
| 服务配置 | SSH 主机、API 端点、认证信息 | 配置文件中 |
| 偏好设置 | TTS 语音、扬声器选择、快捷键 | 用户偏好设置 |
| 凭证信息 | API 密钥、密码、令牌 | 环境变量或密码管理器 |

**传统解决方案的问题**：

- 配置散落在多个文件中，查找困难
- 共享配置时可能泄露敏感信息
- 技能更新时可能覆盖本地配置
- 跨设备使用时配置无法同步

**OpenClaw 的解决方案：配置分层架构**

OpenClaw 将配置信息分为两层：

```
配置分层架构图：

┌─────────────────────────────────────────────────────────┐
│                   共享层（技能/Skills）                   │
│  ├─ 技能定义了工具的_工作方式_                           │
│  ├─ 跨用户、跨设备可共享                                 │
│  ├─ 通过技能仓库统一管理                                 │
│  └─ 包含：API 逻辑、工具实现、默认配置                    │
└─────────────────────────────────────────────────────────┘
                          │
                          ▼
┌─────────────────────────────────────────────────────────┐
│                   本地层（TOOLS.md）                     │
│  ├─ 用户记录你的具体信息                                 │
│  ├─ 特定于你的设置                                       │
│  ├─ 不会随技能更新而丢失                                 │
│  └─ 包含：设备名称、访问地址、个性化偏好                 │
└─────────────────────────────────────────────────────────┘

关键洞察：两层分离，职责清晰
```

**专家思维模型 — 关注点分离（Separation of Concerns）**

软件工程中有一个重要原则：**将不同职责的东西分开**。TOOLS.md 的设计正是这一原则的体现：

| 分离维度 | 技能（Skills） | TOOLS.md |
|----------|----------------|----------|
| **谁使用** | 所有用户 | 特定用户 |
| **谁更新** | 技能开发者 | 用户自己 |
| **更新频率** | 技能发布时更新 | 随时可更新 |
| **共享性** | 可公开共享 | 私密信息 |
| **内容** | 工具的工作原理 | 工具的具体参数 |

**为什么这个设计重要？**

1. **安全性**：本地配置可能包含敏感信息（如 SSH 密码），不应随技能共享
2. **灵活性**：设备名称、访问地址等因人而异，不能统一
3. **可维护性**：技能更新不会覆盖本地个性化配置
4. **协作性**：可以安全地分享技能给他人，而不用担心泄露本地信息

### 1.2 TOOLS.md 的定位

**TOOLS.md 不是什么？**

- **TOOLS.md 不是密码本**：不应存储明文密码或敏感凭证
- **TOOLS.md 不是技能文档**：技能的工作方式应记录在 SKILL.md 中
- **TOOLS.md 不是配置文件**：应用级别的配置应在 openclaw.json 中

**TOOLS.md 是什么？**

TOOLS.md 是**本地环境速查表**，记录那些：
- **特定于你的设置**：只有你知道的信息
- **经常需要查阅**：快速查找的参考信息
- **相对稳定**：不会频繁变更
- **有助于完成任务**：提高效率的辅助信息

**心智模型 — 便利贴 vs 手册**

```
对比：

SKILL.md（技能文档）        TOOLS.md（本地工具）
├─ 工作原理说明              ├─ 设备名称速查
├─ API 使用指南              ├─ 访问地址汇总
├─ 示例代码                  ├─ 配置参数备忘
├─ 故障排除                  ├─ 个性化偏好
└─ 最佳实践                  └─ 环境特定信息

比喻：
SKILL.md = 工具的使用说明书（官方文档）
TOOLS.md = 便利贴（个人速记）
```

### 2.2 各类内容的详细说明

#### 2.2.1 设备信息

**相机设备记录示例**：

```markdown
## 相机设备

| 标识符 | 位置 | 功能特点 | 使用优先级 | 备注 |
|--------|------|----------|------------|------|
| living-room | 客厅主区域 | 180°广角，覆盖范围大 | 1 | 日常监控首选 |
| front-door | 入口 | 运动触发，红外夜视 | 2 | 安防场景 |
| office | 书房 | 桌面角度，适合视频会议 | 3 | 工作相关 |
| bedroom | 卧室 | 隐私区域 | - | 仅紧急情况使用 |
```

**记录要点**：

1. **标识符**：简短、有意义的名称，便于引用
2. **位置**：物理位置或逻辑分组
3. **功能特点**：独特的功能或限制
4. **使用优先级**：当有多个选择时，哪个优先
5. **备注**：任何特殊情况或提醒

#### 2.2.2 SSH 配置

**SSH 配置记录示例**：

```markdown
## SSH 配置

### 主机列表

| 别名 | 地址 | 用户名 | 认证 | 端口 | 用途 |
|------|------|--------|------|------|------|
| home-server | 192.168.1.100 | admin | 密钥 | 22 | 家庭服务器管理 |
| work-dev | dev.company.internal | deploy | 密钥 | 2222 | 开发环境 |
| aws-prod | ec2.prod.amazonaws.com | ubuntu | 密钥 | 22 | 生产服务器 |
| pi-hole | 192.168.1.2 | pi | 密码 | 22 | Pi-hole 管理 |
| backup | backup.server.com | backup | 密钥 | 22022 | 备份服务器 |

### 常用命令别名

```bash
# 这些可以添加到 ~/.ssh/config 中
Host home-server
    HostName 192.168.1.100
    User admin
    IdentityFile ~/.ssh/home_server_key
    ServerAliveInterval 60

Host work-dev
    HostName dev.company.internal
    User deploy
    ProxyJump gateway
    IdentityFile ~/.ssh/work_dev_key
```

### 连接检查

```bash
# 快速检查连接状态
ssh home-server "uptime"
```

**记录要点**：

1. **别名**：简短、易记的名称
2. **地址**：IP 或域名
3. **认证方式**：密钥、密码、MFA 等
4. **端口**：非标准端口特别标注
5. **用途**：为什么连接这个主机
6. **特殊配置**：ProxyJump、端口转发等

### 2.3 不应该记录的内容

TOOLS.md 应该保持简洁，以下内容**不应该**记录：

| 不应该记录 | 原因 | 应该放在哪里 |
|------------|------|--------------|
| 明文密码/密钥 | 安全风险 | 密码管理器或环境变量 |
| 完整的技能文档 | 技能应更新，不应重复 | SKILL.md |
| 临时性信息 | 会造成混淆 | 每日笔记（memory/） |
| 敏感个人数据 | 隐私风险 | 加密存储 |
| 应用配置文件 | 会被覆盖 | openclaw.json 或配置目录 |

**版本标记建议**：

```markdown

> 版本：2.1（2024-01-15）
> 更新者：我自己
> 说明：添加了新的开发服务器配置

## 版本历史

| 版本 | 日期 | 更新内容 | 更新者 |
|------|------|----------|--------|
| v2.1 | 2024-01-15 | 添加 dev.example.com | 我 |
| v2.0 | 2024-01-01 | 重构组织结构 | 我 |
| v1.5 | 2023-12-20 | 添加 TTS 偏好 | 我 |
| v1.0 | 2023-12-01 | 初始版本 | 我 |

**跨设备同步策略**：

| 策略 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| 手动复制 | 简单、安全 | 容易不同步 | 少量设备 |
| 云存储同步 | 自动同步 | 可能有延迟 | 多设备用户 |
| Git 版本控制 | 可追踪历史 | 需要 Git 操作 | 开发者 |
| 符号链接 | 共享同一文件 | 需统一路径 | 路径一致的设备 |

## 第四部分：使用场景与案例

### 4.1 典型使用场景

#### 场景一：远程服务器管理

```
场景描述：
需要在多台服务器上进行运维操作，使用 TOOLS.md 记录连接信息

配置要点：
- 清晰的服务器列表
- 认证方式说明
- 常用操作命令
- 连接状态检查命令

使用收益：
- 快速连接，无需记忆 IP 和端口
- 减少连接错误
- 新增服务器时轻松扩展
```

#### 场景二：多相机监控

```
场景描述：
家中有多个相机，需要 AI 助手访问特定相机

配置要点：
- 每个相机的名称和地址
- 相机的功能特点
- 使用优先级
- 隐私设置

使用收益：
- "打开客厅相机"→ 快速定位到正确的相机
- 语音控制时不会混淆
- 忘记相机名称时可以快速查阅
```

#### 场景三：TTS 语音定制

```
场景描述：
对 TTS 语音有特定偏好，需要记住这些设置

配置要点：
- 语音列表和特点
- 不同场景的语音选择
- 语速、音量等参数
- 输出设备偏好

使用收益：
- "用讲故事的声音读这个"→ 正确识别
- 不同内容类型使用不同语音
- 保持一致的体验
```

## 第五部分：常见问题与故障排除

### 5.1 配置问题诊断

**问题一：设备标识符无法识别**

```
症状：
- SSH 连接时提示 "Host not found"
- TTS 找不到指定的语音

可能原因：
- TOOLS.md 中的标识符与实际设备名称不一致
- 设备已更改名称或配置

排查步骤：
1. 确认设备实际名称：ssh-config 或系统设置
2. 对比 TOOLS.md 中的记录
3. 更新不一致的配置
```

**问题二：配置丢失或被覆盖**

```
症状：
- 之前记录的服务器地址找不到了
- TTS 偏好设置不见了

可能原因：
- 技能更新时 TOOLS.md 被重置
- 工作空间重建
- 手动删除

预防措施：
1. 定期备份 TOOLS.md
2. 使用版本控制
3. 将关键配置添加到 openclaw.json
```

**问题三：多设备配置不同步**

```
症状：
- A 设备上有的配置 B 设备上没有
- 不知道哪个版本是最新的

解决策略：
1. 建立统一的配置模板
2. 使用云存储同步（注意安全）
3. 记录最后更新时间
```

### 5.2 安全最佳实践

```
TOOLS.md 安全检查清单：

□ 不包含明文密码或 API 密钥
□ 不包含完整的银行账户或身份证信息
□ SSH 密钥使用文件路径而非内容
□ 敏感信息使用环境变量引用
□ 定期审查记录的内容
□ 不与他人分享 TOOLS.md
□ 在公共场合使用屏幕遮蔽
```
