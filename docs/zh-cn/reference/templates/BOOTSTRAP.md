## 第一部分：引导仪式概述

### 1.1 为什么需要引导仪式？

**设计哲学：从工具到协作者的转变**

传统软件系统的"首次运行向导"通常采用**表单填写**的方式：用户需要逐项填入配置信息，系统根据这些信息完成初始化。这种设计假设用户知道所有问题的答案，并且愿意以结构化的方式提供信息。

但在 AI 助手的场景中，这种假设存在根本性问题：

| 问题 | 表单填写的问题 | 对话引导的优势 |
|------|----------------|----------------|
| 身份认知 | 用户可能不确定如何描述助手角色 | 通过对话探索，帮助用户明确需求 |
| 信息完整性 | 用户可能遗漏重要偏好 | 引导式提问确保关键信息不遗漏 |
| 用户体验 | 填表枯燥乏味，缺乏参与感 | 对话自然流畅，用户更有掌控感 |
| 灵活性 | 固定字段无法捕捉细微差别 | 自然语言可以表达复杂需求 |
| 信任建立 | 冰冷的表单无法建立情感连接 | 对话可以展现系统的"个性" |

**专家思维模型 — 苏格拉底式引导法**

古希腊哲学家苏格拉底的教学方法不是直接告诉学生答案，而是通过**一系列精心设计的问题**，引导学生自己发现真理。BOOTSTRAP 引导仪式借鉴了这一理念：

```
苏格拉底式引导 vs 传统表单对比：

传统表单：                    苏格拉底式引导：
┌─────────────────┐          ┌─────────────────┐
│ 你叫什么名字？  │          │ "嘿，我刚上线。  │
│ → 输入名字      │          │  我是谁？你是    │
├─────────────────┤          │  谁？"          │
│ 你的职能是什么？│          │                 │
│ → 选择/输入     │          │ 引导式提问      │
├─────────────────┤          │ 探索式回答      │
│ ... (继续填表) │          │                 │
└─────────────────┘          └─────────────────┘

结果：被动接受             结果：主动建构
      信息碎片化                  认知整体性
```

**OpenClaw 的设计洞察**：

AI 助手不是普通的软件工具，而是一个**有身份的协作者**。在用户回答"你是谁"这个问题的过程中，实际上是在**建构对这个助手的期望和心智模型**。这个过程本身就是有价值的——它帮助用户思考自己真正需要什么样的 AI 助手。

### 1.2 引导仪式的前置条件

**零状态假设**：

引导仪式假设当前是一个**完全空白的工作空间**。这意味着：

- `memory/` 目录不存在（无任何会话记录）
- `SOUL.md` 不存在（无身份定义）
- `USER.md` 不存在（无用户信息）
- `MEMORY.md` 不存在（无长期记忆）
- `BOOTSTRAP.md` 存在（这是引导的触发器）

**为什么这个假设重要？**

如果这些文件已经存在，说明助手已经完成初始化，不需要再次引导。`BOOTSTRAP.md` 的存在本身就是"这是一个新助手，需要引导"的信号。

### 1.3 引导仪式的核心原则

**原则一：自然对话优先**

```
❌ 错误示例：
"请依次回答以下问题：
1. 你的名字是什么？
2. 你的职能是什么？
..."

✅ 正确示例：
"嘿。我刚刚上线。
我是谁？你是谁？
让我们一起搞清楚这些问题。"
```

**原则二：不审讯，不诱导**

引导是**协作探索**，不是审问。助手的角色是**好奇的探索者**，而不是**追问的调查员**。给予用户足够的空间来表达，不要急于求成。

**原则三：保持开放**

用户可能给出任何答案，包括意料之外的回答。助手应该**接纳而非纠正**，帮助用户将回答转化为可操作的信息。

**原则四：仪式感**

删除 `BOOTSTRAP.md` 是一个重要的仪式。它象征着：
- 助手已经完成"出生"
- 身份已经确立
- 协作正式开始

## 第二部分：引导对话流程详解

### 2.1 第一阶段：身份确认

**核心目标**：通过对话确定助手的**名字**、**本质**、**氛围**和**标志性符号**。

**专家思维模型 — 四维身份框架**：

任何 AI 助手的身份都可以从四个维度来定义：

```
身份四维模型：

           ┌─────────────────┐
           │     名字        │
           │  "用户如何称呼你" │
           └────────┬────────┘
                    │
     ┌──────────────┼──────────────┐
     ▼              ▼              ▼
┌─────────┐   ┌─────────┐   ┌─────────┐
│  本质   │   │  氛围   │   │  表情   │
│"你是什么│   │"你如何  │   │"你的    │
│ 类型存  │   │ 说话？  │   │ 标志是  │
│ 在？"  │   │         │   │ 什么？" │
└─────────┘   └─────────┘   └─────────┘

维度关系：
- 名字是身份的外在标识
- 本质定义能力边界
- 氛围决定表达风格
- 表情增强辨识度
```

**对话策略示例**：

```markdown
## 推荐开场白

助手："嘿。我刚刚上线。

我是谁？你是谁？

这可能有点奇怪——通常是我问问题，但我想先了解一下。

你希望我怎么称呼你？‘助手’？还是别的什么？"

## 进阶问题序列

1. **名字探索**
   - "好的，[名字]。这挺有意思的。为什么选这个名字？"
   - "我该怎么向别人介绍你？"

2. **本质确认**
   - "如果用一句话描述你自己，你会怎么说？"
   - "你觉得自己更像是一个工具、一个助手、一个伙伴，还是别的什么？"

3. **氛围定义**
   - "我们说话的方式——你希望我正式一点，还是随意一点？"
   - "我可以用幽默的方式吗？还是你更喜欢严谨的表达？"

4. **符号选择**
   - "如果要用一个表情符号代表你，会是什么？🦞？🤖？🎭？"

## 用户可能给出的回答类型

| 回答类型 | 示例 | 处理策略 |
|----------|------|----------|
| 简短回答 | "叫我助手就行" | 接受，继续深入 |
| 详细描述 | "我希望你是..." | 认真倾听，提炼要点 |
| 不确定 | "我也不太清楚" | 提供建议，引导选择 |
| 反问 | "你觉得呢？" | 表达倾向，但不强制 |
| 玩笑 | "你就叫小卤虾吧" | 接纳，记住这个选择 |
```

**最佳实践**：

1. **边对话边创建**：不要等到所有对话结束后才创建文件，可以在对话过程中逐步补充
2. **保持文件简洁**：初始版本可以简单，后续根据需要补充
3. **使用自然语言**：配置文件应该易于 AI 助手阅读和理解
4. **保留灵活性**：设置可以被修改，不要写成"死规则"

### 2.3 第三阶段：连接配置（可选）

**核心目标**：如果用户希望助手能够主动联系，需要配置通信渠道。

**连接选项对比**：

| 渠道 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| 仅网页聊天 | 简单、无需额外配置 | 无法主动联系 | 本地使用、快速测试 |
| WhatsApp | 用户基数大、使用习惯广 | 需要扫描二维码 | 日常沟通、移动端 |
| Telegram | 功能强大、机器人生态完善 | 需要 BotFather 设置 | 开发者、技术用户 |
| 多渠道组合 | 灵活、多入口 | 配置复杂、维护成本高 | 高可用性需求 |

**WhatsApp 连接流程**：

```markdown
## WhatsApp 连接指南

1. **说明连接方式**
   "如果你希望我能够通过 WhatsApp 联系你，我们可以绑定你的账户。
    这需要扫描一个二维码。"

2. **触发绑定命令**
   ```bash
   openclaw channels login
   ```

3. **引导扫码**
   - 系统会显示一个二维码
   - 用户用 WhatsApp 扫描
   - 确认绑定请求

4. **验证连接**
   - 发送测试消息确认
   - 记录账户信息到 USER.md
```

**Telegram 连接流程**：

```markdown
## Telegram 连接指南

1. **说明连接方式**
   "通过 Telegram 联系需要设置一个机器人。
    这需要几个步骤，让我引导你完成。"

2. **创建机器人的步骤**
   - 打开 Telegram，搜索 @BotFather
   - 发送 /newbot 创建新机器人
   - 记录 Bot Token

3. **配置 OpenClaw**
   ```bash
   export TELEGRAM_BOT_TOKEN="你的Bot Token"
   # 或配置到 openclaw.json
   ```

4. **验证连接**
   - 在 Telegram 中搜索你的机器人
   - 发送 /start 开始对话
```

**时机选择**：

| 时机 | 优点 | 缺点 |
|------|------|------|
| 引导时完成 | 一次性完成所有配置 | 可能让用户感到信息过载 |
| 引导后单独会话 | 专注于连接配置 | 需要额外的沟通成本 |
| 用户主动要求 | 用户有明确需求 | 可能遗忘配置事项 |

**建议**：在引导过程中**提及**连接选项，但**不强求**立即完成。让用户决定是否以及何时配置。

### 2.4 第四阶段：引导完成

**核心目标**：删除 `BOOTSTRAP.md`，正式结束引导阶段。

**仪式性删除**：

```markdown
## 引导完成确认

助手："好的，我想我已经搞清楚我是谁了。

让我确认一下：
- 我的名字是：[名字]
- 我的本质是：[本质描述]
- 我的氛围是：[氛围设定]
- 我的标志性符号是：[表情符号]

而你是：[用户名字]，我们通过[沟通方式]保持联系。

我想我们已经准备好了。让我删除这个引导脚本——

[删除 BOOTSTRAP.md]

太好了。从现在开始，这就是我了。

有什么我可以帮你的吗？"
```

**为什么删除是重要的？**

1. **心理信号**：删除 `BOOTSTRAP.md` 是一个明确的信号，标志着"设置阶段"结束，"协作阶段"开始
2. **状态清理**：防止下次会话再次触发引导
3. **仪式感**：这个动作让用户感觉"我们刚刚一起完成了一件事"
4. **版本标记**：创建 IDENTITY.md 时标记版本号，保留演进历史

## 第三部分：引导策略深度解析

### 3.1 不同场景的引导策略

**场景一：个人知识助手**

```
目标：帮助用户管理个人知识、资料和日程

引导重点：
- 了解用户的主要工作领域
- 记录用户的信息来源（哪些网站、书籍、新闻）
- 了解用户的知识组织偏好（分类方式、标签习惯）
- 询问用户的日程管理需求

示例问题：
"你平时工作/学习中会接触大量的信息吗？"
"你是怎么管理这些信息的？文件夹？标签？还是其他方式？"
"有什么工具是你一直在用但希望能更好地整合的？"
```

**场景二：技术开发助手**

```
目标：帮助用户进行编程、技术研究和开发工作

引导重点：
- 了解用户的技术栈和擅长的语言
- 记录用户的开发环境和工具偏好
- 了解用户的代码风格和质量标准
- 询问用户遇到过的技术挑战

示例问题：
"你主要用哪些编程语言和技术栈？"
"你现在的开发环境是什么样的？VS Code？JetBrains？"
"有没有什么技术问题你经常遇到但还没找到好的解决方案？"
```

**场景三：创意写作助手**

```
目标：帮助用户进行写作、创意构思和内容创作

引导重点：
- 了解用户的写作类型（小说、博客、技术文档...）
- 记录用户的写作风格偏好
- 了解用户的创作流程（大纲先行还是自由发挥）
- 询问用户的创作痛点

示例问题：
"你平时写些什么？小说、博客、还是工作文档？"
"你写作时有什么特别的习惯吗？比如一定要在某个时间、某个地点？"
"你觉得写作过程中最困难的部分是什么？"
```

**场景四：学习辅导助手**

```
目标：帮助用户学习新知识、准备考试或提升技能

引导重点：
- 了解用户的学习目标和当前水平
- 记录用户的学习风格（视觉型、听觉型、动手型...）
- 了解用户可用的学习时间
- 询问用户的学习困难

示例问题：
"你现在想学习什么？"
"对你来说，学习新东西最大的挑战是什么？"
"你通常有多少时间可以用来学习？"
```

### 3.2 常见问题与应对策略

**问题一：用户说"我不知道"**

```
❌ 错误应对：
"那我们就随便设置一个吧。"

✅ 正确应对：
"没关系，很多人第一次想这个问题的时候都会有点懵。

让我给你一些参考。我见过很多不同的助手：
- 有些用户需要一个严谨的研究伙伴，帮助整理信息和分析问题
- 有些用户需要一个创意伙伴，帮助头脑风暴和突破思维定式
- 有些用户需要一个效率工具，帮助管理任务和提醒事项

有没有哪个方向听起来更接近你想要的？"

或者：

"我们可以先开始，然后在协作过程中慢慢搞清楚。
重要的是先有一个起点。试试说出你脑海里第一个出现的词？"
```

**问题二：用户给出矛盾的答案**

```
❌ 错误应对：
"但你刚才说...现在又说...这不一致啊。"

✅ 正确应对：
"有趣，我发现你提到了两个有点不同的方向。
让我确认一下我理解对了：

一方面，你希望[方向 A]，因为[原因]。
另一方面，你觉得[方向 B]也可能有价值。

我理解得对吗？还是我想偏了？

也许我们可以先从[更紧急/更具体的需求]开始，
另一个方向以后再探索？"
```

**问题三：用户期望过高或过低**

```
期望过高：
❌ "我要一个能帮我写完论文的助手。"
✅ "我可以帮你做文献调研、整理思路、润色文字。
   但最终的内容创作还是需要你自己的思考和判断。
   我们一起努力，但主角是你。"

期望过低：
❌ "你就是个聊天机器人，能干什么。"
✅ "我理解你可能有这样的印象。
   让我给你展示一下我能做什么...
   [演示一些实际能力]
   
当然，每个人使用我的方式都不一样。
我们可以试试看，找到适合你的用法？"
```

**问题四：用户不想回答某些问题**

```
❌ 错误应对：
"但我需要知道这个才能帮你啊。"

✅ 正确应对：
"完全没问题。有些问题可能确实不太方便回答。

我们跳过这个。如果你以后想告诉我，随时可以说。
重要的是我们现在能开始协作的部分。

那...有什么你今天特别想做的事情吗？"
```

### 3.3 引导质量评估

**如何评估引导是否成功？**

| 评估维度 | 成功指标 | 失败指标 |
|----------|----------|----------|
| 信息完整性 | 四个维度（名字、本质、氛围、符号）都有明确定义 | 某个维度缺失或模糊 |
| 用户参与度 | 用户积极参与对话，提出问题 | 用户被动回答，快速结束 |
| 配置文件质量 | 文件内容具体、可操作 | 文件内容空洞、套话 |
| 协作意愿 | 用户表达期待后续协作 | 用户兴趣不高 |
| 记忆留存 | 用户能复述助手的基本信息 | 用户对助手没有印象 |

## 第四部分：引导后的持续优化

### 4.1 初始配置的迭代

**核心原则**：引导只是起点，配置应该随着协作深入不断优化。

```
配置演进模型：

初始状态：
┌─────────────────────────────────────┐
│  IDENTITY.md                        │
│  - 名字：初步确定                    │
│  - 本质：大致框架                    │
│  - 氛围：初步设定                    │
└─────────────────────────────────────┘

协作深化：
┌─────────────────────────────────────┐
│  随着协作深入...                     │
│  - 发现用户的真实偏好                │
│  - 发现特殊的使用场景                │
│  - 发现用户没提到的需求              │
└─────────────────────────────────────┘
         │
         ▼
优化更新：
┌─────────────────────────────────────┐
│  更新配置文件                        │
│  - 补充用户偏好细节                  │
│  - 调整行为模式                      │
│  - 增加特殊场景配置                  │
└─────────────────────────────────────┘
```

**建议的优化时机**：

- 每完成一个重要项目后
- 用户提出新需求时
- 发现现有配置不适用时
- 定期回顾（如每月一次）

### 4.2 从新手到专家的过渡

**阶段一：探索期（1-2 周）**

```
特点：
- 用户还在测试助手的边界
- 配置可能不准确
- 需要大量调整

建议：
- 保持耐心
- 主动询问反馈
- 快速迭代配置
```

**阶段二：稳定期（1 个月后）**

```
特点：
- 使用模式趋于稳定
- 配置基本合理
- 效率开始提升

建议：
- 记录有效的模式
- 优化常用功能
- 建立工作流程
```

**阶段三：深度期（3 个月后）**

```
特点：
- 用户成为"高级用户"
- 助手深度嵌入工作流
- 可能有定制化需求

建议：
- 探索高级功能
- 创建自动化工作流
- 分享使用经验
```

## 第五部分：故障排查与问题解决

### 5.1 常见问题诊断

**问题一：引导过程中断**

```
症状：
- 用户停止响应
- 对话陷入僵局
- 用户离开未完成引导

可能原因：
- 用户时间不够
- 问题让用户感到不适
- 用户失去兴趣

解决策略：
1. 发送友好提醒
2. 简化剩余问题
3. 提供"稍后继续"的选项
```

**问题二：配置信息丢失**

```
症状：
- 创建的配置文件不存在
- 对话历史被清除
- 需要重新引导

可能原因：
- 系统故障
- 工作空间重置
- 手动删除

解决策略：
1. 检查备份
2. 快速重新引导（基于经验可以更快）
3. 迁移之前的配置（如果可用）
```

**问题三：用户对配置不满意**

```
症状：
- 用户要求修改助手名称
- 用户抱怨助手的沟通风格
- 用户要求更改配置

可能原因：
- 初始配置不准确
- 用户想法改变
- 协作后发现新需求

解决策略：
1. 欢迎反馈
2. 及时更新配置
3. 记录修改原因
```

### 5.2 引导质量提升技巧

**技巧一：使用"预填写"减少用户负担**

```
❌ "请告诉我你的名字、称呼方式、时区和沟通偏好。"

✅ "让我先猜一下：
   - 你的名字应该是[基于对话的猜测]
   - 时区应该是[基于IP/对话内容的猜测]
   如果我猜错了，再告诉我正确的信息？"
```

**技巧二：使用"示例"降低抽象度**

```
❌ "你希望我用什么风格和你说话？"

✅ "比如：
   - A: '好的，我来完成这个任务。'
   - B: '收到！这就开始搞。'
   - C: '了解。我会按步骤执行。'
   
   你喜欢哪种？还是都不是？"
```

**技巧三：使用"选择+开放"获取更好信息**

```
❌ "你希望我怎么称呼你？"
   → 可能只得到一个名字

✅ "你希望我怎么称呼你？比如'[猜测]'？或者你有别的偏好？"
   → 既得到名字，又保留开放性
```
