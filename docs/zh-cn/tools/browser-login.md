# 浏览器登录 + X/Twitter 发帖

## 手动登录（推荐方式）

当网站需要登录时，请在**宿主**浏览器配置文件（openclaw 浏览器）中**手动登录**。

**不要**把你的凭据交给模型。自动化登录通常会触发反机器人防护机制，可能导致账户被锁定。

返回浏览器主文档：[浏览器工具](/tools/browser)

## 为什么需要手动登录

1. **安全性**：避免凭据泄露给 AI 模型
2. **稳定性**：自动登录容易触发验证码和账户风控
3. **合规性**：许多网站的服务条款禁止自动化登录
4. **持久性**：手动登录的会话通常更持久稳定

## 使用的 Chrome 配置文件

OpenClaw 控制一个**专用的 Chrome 配置文件**（名为 `openclaw`，橙色主题 UI）。这与你日常使用的浏览器配置文件是分开的。

**两种简单的访问方式：**

1. **让智能体打开浏览器**，然后自己登录
2. **通过 CLI 打开**：

```bash
openclaw browser start
openclaw browser open https://x.com
```

如果你有多个配置文件，使用 `--browser-profile <名称>` 参数（默认是 `openclaw`）。

## X/Twitter：推荐工作流

| 操作类型 | 推荐方式 | 说明 |
|----------|----------|------|
| 阅读/搜索/查看帖子 | **bird** CLI 技能 | 无需浏览器，稳定可靠 |
| 发布更新 | **宿主**浏览器 | 需要手动登录 |

**bird CLI 仓库**：https://github.com/steipete/bird

## 沙箱与宿主浏览器访问

沙箱化的浏览器会话**更容易**触发机器人检测。对于 X/Twitter（及其他严格的网站），优先使用**宿主**浏览器。

### 配置宿主浏览器访问

如果智能体处于沙箱模式，浏览器工具默认使用沙箱。要允许宿主控制：

```json5
{
  agents: {
    defaults: {
      sandbox: {
        mode: "non-main",
        browser: {
          allowHostControl: true,
        },
      },
    },
  },
}
```

然后指定宿主浏览器：

```bash
openclaw browser open https://x.com --browser-profile openclaw --target host
```

或者对发帖的智能体禁用沙箱。

## 登录最佳实践

| 实践 | 说明 |
|------|------|
| 使用专用配置文件 | 不要在个人浏览器配置文件中操作 |
| 定期检查会话 | 某些网站会定期清除会话 |
| 启用双因素认证 | 保护账户安全 |
| 记录登录状态 | 知道哪些网站已登录 |

## 常见问题

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| 登录后立即被登出 | Cookie 未正确保存 | 检查用户数据目录配置 |
| 触发验证码 | 自动化行为被检测 | 使用手动登录，放慢操作速度 |
| 账户被临时锁定 | 频繁的自动化请求 | 等待解锁后使用手动登录 |
| 无法访问宿主浏览器 | 沙箱限制 | 配置 `allowHostControl: true` |

## 安全提醒

- **永远不要**在聊天中分享密码
- **永远不要**让 AI 模型填写登录表单
- 将登录状态视为敏感信息
- 定期审查已登录的会话
