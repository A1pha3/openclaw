## 💡 消息处理流程

### 类比：消息处理就像"智能客服系统"

```
┌─────────────────────────────────────────────────────────────────────────┐
│                    智能客服系统类比                                     │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  现实场景：客服中心                                                     │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │  顾客发消息 ──► 前台接待 ──► 检查重复 ──► 合并快速消息            │   │
│  │                   │                                           │   │
│  │                   ▼                                           │   │
│  │              业务员处理 ──► 分段回复 ──► 发送给顾客              │   │
│  │                                                                 │   │
│  │  处理要点：                                                      │   │
│  │  • 同一消息不处理两次（去重）                                    │   │
│  │  • 快速连续消息合并处理（防抖）                                  │   │
│  │  • 长回复分段发送（流式传输）                                    │   │

│  │                   │                                           │   │
│  │                   ▼                                           │   │
│  │              智能体运行 ──► 流式传输 ──► 渠道发送                 │   │
│  │                                                                 │   │
│  │  处理要点：                                                      │   │
│  │  • 入站去重：渠道重连时不重复处理                                │   │
│  │  • 入站防抖：快速消息合并处理                                    │   │
│  │  • 分块流式：长回复分段发送                                      │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘

## 🔁 入站去重

### 为什么需要去重？

**问题场景**：渠道重连后可能会重复投递同一消息

```
┌─────────────────────────────────────────────────────────────────────────┐
│                    消息重复问题                                         │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  没有去重：                                                             │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │  用户发送 "Hello"                                               │   │
│  │     │                                                           │   │
│  │     ├─► 第一次投递 ──► AI 回复 "Hi!"                            │   │
│  │     │                                                           │   │
│  │     └─► 重连后重复投递 ──► AI 再次回复 "Hi!" ❌                 │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
│  有去重：                                                               │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │  用户发送 "Hello"                                               │   │
│  │     │                                                           │   │
│  │     ├─► 第一次投递 ──► 缓存消息 ID ──► AI 回复 "Hi!" ✓          │   │
│  │     │                                                           │   │
│  │     └─► 重连后重复投递 ──► 检测到已处理 ──► 丢弃 ✓              │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### 去重机制

```
缓存键: channel:account:peer:session:messageId

┌─────────────────────────────────────────────────────────────────────────┐
│                    去重缓存结构                                         │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  消息到达                                                                │
│     │                                                                   │
│     ▼                                                                   │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │  计算缓存键                                                      │   │
│  │  {channel}:{account}:{peer}:{session}:{messageId}               │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│     │                                                                   │
│     ▼                                                                   │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │  检查缓存                                                        │   │
│  │                                                                 │   │
│  │  键存在？                                                         │   │
│  │  ├─ YES ──► 丢弃消息（已处理）                                  │   │
│  │  └─ NO ──► 继续处理 → 存入缓存                                 │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
│  注意：缓存是短期的，自动过期                                          │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

## ⏱️ 入站防抖

### 防抖机制

**目的**：将同一发送者的快速连续消息合并为单个智能体轮次

```
┌─────────────────────────────────────────────────────────────────────────┐
│                    防抖机制演示                                         │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  时间线（防抖窗口: 2 秒）                                               │
│                                                                         │
│  消息1: "Hello"      ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━        │
│                         ▲                                              │
│                         │                                              │
│  消息2: "How are you?" ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━     │
│                         ▲                                              │
│                         │                                              │
│  消息3: "Help me"     ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━     │
│                         │                                              │
│                         ▼                                              │
│                    防抖窗口结束                                        │
│                         │                                              │
│                         ▼                                              │
│  合并处理: "Hello\nHow are you?\nHelp me" ──► AI 回复                  │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### 防抖配置

```json5
{
  messages: {
    inbound: {
      debounceMs: 2000,        // 全局默认（毫秒）
      byChannel: {
        whatsapp: 5000,        // WhatsApp: 5 秒
        slack: 1500,           // Slack: 1.5 秒
        discord: 1500,         // Discord: 1.5 秒
        telegram: 2000,        // Telegram: 2 秒
      },
    },
  },
}
```

### 防抖规则

| 规则 | 说明 |
|------|------|
| **适用范围** | 按渠道 + 会话 |
| **消息类型** | 仅纯文本 |
| **例外** | 媒体/附件立即刷新 |
| **例外** | 控制命令绕过防抖 |
| **回复线程** | 使用最近消息的线程 ID |

## 📦 正文处理

### 正文类型

```
┌─────────────────────────────────────────────────────────────────────────┐
│                    消息正文类型                                         │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  原始消息: "请帮我 /status 检查状态"                                    │
│     │                                                                   │
│     ▼                                                                   │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │  CommandBody / RawBody                                          │   │
│  │  "请帮我 /status 检查状态"                                       │   │
│  │  用途: 命令解析、指令识别                                        │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│     │                                                                   │
│     ▼                                                                   │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │  Body（处理后）                                                 │   │
│  │  可能包含：                                                      │   │
│  │  • 渠道信封（发送者信息等）                                      │   │
│  │  • 历史上下文包装器                                              │   │
│  │  • [Chat messages since last reply]                            │   │
│  │  • [Current message - respond to this]                          │   │
│  │  用途: 发送给智能体的完整提示                                    │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### 历史上下文包装

对于提供历史记录的渠道：

```
[Chat messages since your last reply - for context]
User: 之前的消息1
Assistant: AI回复1
User: 之前的消息2
[Current message - respond to this]
User: 当前消息 ← AI 响应这个
```

### 非直接聊天的特殊处理

**群组/渠道/房间**：当前消息正文会加上发送者标签

```
群组消息 → [Alice] 请帮我检查状态
历史记录 → [Bob] 之前的消息
         → [Carol] 另一条消息
```

这使智能体提示中的实时消息和队列/历史消息保持一致。

### 历史缓冲区配置

```json5
{
  messages: {
    groupChat: {
      historyLimit: 50,        // 全局默认
    },
  },
  channels: {
    slack: {
      historyLimit: 100,       // Slack 覆盖
    },
    telegram: {
      accounts: {
        mybot: {
          historyLimit: 0,     // 禁用历史
        },
      },
    },
  },
}
```

## 🚀 队列和后续消息

### 队列模式

当智能体正在运行时，新消息可以：

| 模式 | 行为 | 适用场景 |
|------|------|----------|
| **interrupt** | 立即中断当前任务 | 紧急命令 |
| **steer** | 重定向当前任务 | 修正方向 |
| **followup** | 追加到当前任务 | 补充说明 |
| **collect** | 收集用于下一轮 | 批量处理 |

### 队列配置

```json5
{
  messages: {
    queue: {
      mode: "followup",       // 默认模式
      backlogMode: "enqueue", // 积压模式
      byChannel: {
        whatsapp: {
          mode: "interrupt",
        },
      },
    },
  },
}
```

**详情**：参见 [队列](/concepts/queue)

## 📡 流式传输和分块

### 分块流式传输

```
┌─────────────────────────────────────────────────────────────────────────┐
│                    分块流式传输                                         │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  模型输出流:                                                             │
│  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━     │
│                                                                         │
│  分块 1: "这是一个完整的代码示例："     ──► 发送                         │
│  分块 2: "\n```python\ndef hello():"     ──► 发送（代码块开始）        │
│  分块 3: "    print('Hello')\n```\n"    ──► 发送（代码块结束）        │
│  分块 4: "你可以运行它。"                 ──► 发送                       │
│                                                                         │
│  优势：                                                                  │
│  ✅ 避免拆分代码块                                                     │
│  ✅ 遵守渠道文本限制                                                   │
│  ✅ 提供即时反馈                                                       │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### 关键配置

| 配置项 | 说明 |
|--------|------|
| `blockStreamingDefault` | 开关（on\|off） |
| `blockStreamingBreak` | 断点位置（text_end\|message_end） |
| `blockStreamingChunk` | 分块大小（minChars\|maxChars） |
| `blockStreamingCoalesce` | 批处理配置 |
| `humanDelay` | 块间延迟（拟人化） |

**详情**：参见 [流式传输](/concepts/streaming)

## 🧠 推理可见性

### 推理模式

| 模式 | 说明 | Token 计费 |
|------|------|------------|
| **off** | 隐藏推理 | 计入 |
| **on** | 显示推理 | 计入 |
| **stream** | 流式显示推理 | 计入 |

**重要**：推理内容始终计入 token 使用量，无论是否可见。

### 控制命令

```
/reasoning on     # 显示推理
/reasoning off    # 隐藏推理
/reasoning stream # 流式显示推理（Telegram 草稿气泡）
```

## 🐛 故障排查

### 常见问题

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| 消息重复响应 | 去重缓存失效 | 检查渠道重连频率 |
| 快速消息丢失 | 防抖窗口太长 | 减少 `debounceMs` |
| 长回复被截断 | 渠道限制 | 启用分块流式传输 |
| 历史混乱 | historyLimit 过高 | 减少历史条目 |
| 命令不响应 | 防抖捕获命令 | 确保命令绕过防抖 |

## 🎯 最佳实践

### 按渠道配置

```json5
{
  messages: {
    inbound: {
      byChannel: {
        // 桌面渠道：快速响应
        slack: { debounceMs: 1000 },
        discord: { debounceMs: 1000 },

        // 移动渠道：稍长等待
        whatsapp: { debounceMs: 3000 },
        telegram: { debounceMs: 2000 },

        // 邮件：批量处理
        email: { debounceMs: 10000 },
      },
    },
  },
}
```

### 性能优化

| 实践 | 说明 |
|------|------|
| **合理防抖** | 平衡响应速度和合并效率 |
| **历史限制** | `historyLimit: 20-50` 足够 |
| **分块流式** | 长回复必须启用 |
| **推理隐藏** | 生产环境通常用 `off` |
