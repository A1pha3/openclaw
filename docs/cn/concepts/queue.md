## 💡 为什么需要队列？

### 类比：队列就像"餐厅排队"

```
┌─────────────────────────────────────────────────────────────────────────┐
│                    队列类比说明                                        │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  没有队列的混乱场景：                                                    │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │  顾客 A ──┐                                                        │   │
│  │  顾客 B ──┼──► 同时冲向服务员                                   │   │
│  │  顾客 C ──┘                                                        │   │
│  │                                                                 │   │
│  │  问题：                                                          │   │
│  │  • 服务员不知道先服务谁                                         │   │
│  │  • 订单可能混淆                                                 │   │
│  │  • 资源竞争（笔、纸）                                           │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
│  有队列的有序场景：                                                      │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │  顾客 A ──► 队列 ──► 服务 ──► 完成                               │   │
│  │  顾客 B ──►   │         │         │                             │   │
│  │  顾客 C ──►   │         │         │                             │   │
│  │                │         │         │                             │   │
│  │  优势：                                                          │   │
│  │  ✅ 先到先服务                                                    │   │
│  │  ✅ 订单不会混淆                                                 │   │
│  │  ✅ 资源安全使用                                                 │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### OpenClaw 队列的作用

| 价值 | 说明 |
|------|------|
| **防止冲突** | 避免多个智能体运行同时修改同一会话 |
| **资源保护** | 保护会话文件、日志、CLI stdin 等共享资源 |
| **速率控制** | 降低上游 API 速率限制的可能性 |
| **安全并行** | 允许不同会话的安全并行处理 |

## 🏗️ 队列架构

### 工作原理

```
┌─────────────────────────────────────────────────────────────────────────┐
│                    队列处理流程                                        │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  入站消息（多渠道）                                                      │
│     │                                                                   │
│     ▼                                                                   │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │  会话通道（session:<key>）                                        │   │
│  │  • 保证每个会话只有一个活动运行                                  │   │
│  │  • 并发限制：每个会话 1 个                                        │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│     │                                                                   │
│     ▼                                                                   │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │  全局通道（main / subagent / cron）                              │   │
│  │  • 控制整体并行度                                                │   │
│  │  • 并发限制：maxConcurrent（main 默认 4）                        │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│     │                                                                   │
│     ▼                                                                   │
│  智能体运行（按队列顺序执行）                                           │
│     │                                                                   │
│     ▼                                                                   │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │  通知系统                                                        │   │
│  │  • 等待超过 2 秒时发出通知                                      │   │
│  │  • 输入指示器立即触发（用户体验不受影响）                         │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### 通道类型

| 通道 | 默认并发 | 用途 |
|------|----------|------|
| `main` | 4 | 入站自动回复 + 主心跳 |
| `subagent` | 8 | 子智能体任务 |
| `cron` | 按配置 | 定时任务 |
| 自定义 | 按配置 | 特定用途 |

## 🔄 队列模式

### 模式对比

```
┌─────────────────────────────────────────────────────────────────────────┐
│                    队列模式对比                                         │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  collect（默认，收集合并）                                             │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │  消息 1 ─┐                                                       │   │
│  │  消息 2 ─┼──► 合并为单个后续轮次                                │   │
│  │  消息 3 ─┘                                                       │   │
│  │                                                                 │   │
│  │  适用：快速连续的消息，避免多次回复                             │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
│  followup（后续轮次）                                                    │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │  消息 1 ──► 运行 1 ──► 回复 1                                   │   │
│  │  消息 2 ───────► 等待 ──► 运行 2 ──► 回复 2                     │   │
│  │  消息 3 ──────────────────► 等待 ──► 运行 3 ──► 回复 3           │   │
│  │                                                                 │   │
│  │  适用：需要分别处理每条消息                                     │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
│  steer（引导注入）                                                        │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │  当前运行中...                                                   │   │
│  │  新消息 ──► 注入当前运行（下一个工具边界后取消待处理工具）         │   │
│  │                                                                 │   │
│  │  适用：需要立即影响当前响应                                     │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
│  steer-backlog（引导+保留）                                             │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │  新消息 ──► 注入当前运行 + 保留用于后续轮次                    │   │
│  │                                                                 │   │
│  │  注意：可能导致重复回复                                         │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### 模式详解

| 模式 | 说明 | 使用场景 |
|------|------|----------|
| **collect** | 合并为单个后续轮次 | 快速连续的消息（默认） |
| **followup** | 每条消息独立轮次 | 需要分别处理 |
| **steer** | 注入当前运行 | 需要立即影响当前响应 |
| **steer-backlog** | 注入 + 保留 | 需要当前影响 + 后续 |
| **interrupt** | 中断当前运行 | 紧急情况 |

## ⚙️ 配置选项

### 基础配置

```json5
{
  messages: {
    queue: {
      mode: "collect",           // 队列模式
      debounceMs: 1000,           // 防抖延迟
      cap: 20,                   // 最大排队消息数
      drop: "summarize",         // 溢出策略
      byChannel: {
        discord: "collect",
        telegram: "followup"
      }
    }
  }
}
```

### 并发配置

```json5
{
  agents: {
    defaults: {
      maxConcurrent: 4,           // 全局并发限制
      queue: {
        main: 4,                  // main 通道并发
        subagent: 8,              // subagent 通道并发
        cron: 2                   // cron 通道并发
      }
    }
  }
}
```

### 队列选项

| 选项 | 默认值 | 说明 |
|------|--------|------|
| `mode` | `"collect"` | 队列模式 |
| `debounceMs` | `1000` | 等待静默的时间（毫秒） |
| `cap` | `20` | 每个会话的最大排队消息数 |
| `drop` | `"summarize"` | 溢出策略 |

### 溢出策略

```
┌─────────────────────────────────────────────────────────────────────────┐
│                    溢出策略                                             │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  old（丢弃旧消息）                                                      │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │  [消息 1, 消息 2, ..., 消息 20] → 新消息                         │   │
│  │                                   │                               │   │
│  │                                   ▼                               │   │
│  │  丢弃消息 1，保留新消息                                         │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
│  new（丢弃新消息）                                                      │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │  [消息 1, 消息 2, ..., 消息 20] → 新消息                         │   │
│  │                                   │                               │   │
│  │                                   ▼                               │   │
│  │  拒绝新消息，保留旧消息                                         │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
│  summarize（总结，默认）                                                │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │  超过 cap 时，丢弃的消息被总结为要点                              │   │
│  │  然后作为合成的后续提示注入                                     │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

## 💬 会话内命令

### `/queue` 命令

```
# 查看当前模式
/queue

# 设置队列模式
/queue collect
/queue followup
/queue steer
/queue steer-backlog

# 组合选项
/queue collect debounce:2s cap:25 drop:summarize

# 重置为默认
/queue default
/queue reset
```

### 选项语法

```
debounce:<时间>    # 例如：debounce:500ms, debounce:2s
cap:<数字>         # 例如：cap:15, cap:30
drop:<策略>        # 例如：drop:old, drop:new, drop:summarize
```

## 📊 范围和保证

### 适用范围

```
✅ 适用于：
• 所有使用 Gateway 网关回复管道的入站渠道
• WhatsApp 网页版、Telegram、Slack、Discord
• Signal、iMessage、网页聊天等

❌ 不适用于：
• 手动触发的 CLI 运行
• 某些特殊内部任务
```

### 保证机制

| 保证 | 说明 |
|------|------|
| **会话隔离** | 每个会话同时只有一个智能体运行 |
| **顺序处理** | 同一会话内按消息顺序处理 |
| **并发控制** | 整体并行度受 `maxConcurrent` 限制 |
| **无外部依赖** | 纯 TypeScript + promises 实现 |

## 🐛 故障排查

### 常见问题

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| 命令"卡住" | 队列正在等待 | 启用详细日志查看状态 |
| 消息丢失 | cap 太小 | 增加 `cap` 值 |
| 响应延迟 | 并发太低 | 增加 `maxConcurrent` |
| 重复回复 | 使用了 `steer-backlog` | 改用 `collect` |

### 调试技巧

```bash
# 启用详细日志
openclaw gateway --verbose

# 查找队列日志
grep "queued for" /tmp/openclaw/openclaw-*.log

# 检查并发设置
openclaw config get agents.defaults.maxConcurrent
openclaw config get messages.queue
```

### 日志解读

| 日志 | 含义 |
|------|------|
| `queued for ...ms` | 等待时间 |
| `Queue depth: N` | 当前队列深度 |
| `Channel main: M/N` | 通道使用情况 |
