## 💡 为什么需要会话管理？

### 类比：会话就像对话笔记本

想象 OpenClaw 是一个**智能笔记本管理员**：

```
┌─────────────────────────────────────────────────────────────────┐
│                    会话管理类比                                  │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  传统方式（无会话管理）:                                       │
│  ┌─────────────────────────────────────────────────────┐       │
│  │  一个巨大的笔记本，所有对话混在一起                      │       │
│  │  - Alice 和 Bob 的消息混在一起                          │       │
│  │  - 新消息需要翻遍所有历史                               │       │
│  │  - 隐私泄露风险高                                       │       │
│  └─────────────────────────────────────────────────────┘       │
│                                                                 │
│  OpenClaw 方式（智能会话管理）:                                │
│  ┌──────────────────┐  ┌──────────────────┐  ┌──────────────┐  │
│  │ Alice 的笔记本   │  │ Bob 的笔记本     │  │ 工作群组     │  │
│  │ (私聊会话)        │  │ (私聊会话)        │  │ (群组会话)   │  │
│  └──────────────────┘  └──────────────────┘  └──────────────┘  │
│                                                                 │
│  优势:                                                          │
│  ✅ 隔离：每个对话独立                                        │
│  ✅ 上下文：只加载相关历史                                    │
│  ✅ 安全：Alice 看不到 Bob 的对话                             │
│  ✅ 效率：快速定位和检索                                      │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 会话管理的核心价值

| 价值 | 说明 | 示例 |
|------|------|------|
| **上下文连续性** | AI 记住之前说过什么 | "我们刚才讨论的项目" |
| **隐私隔离** | 用户数据不交叉 | Alice 的秘密不会泄露给 Bob |
| **资源优化** | 只加载需要的对话历史 | 减少 Token 使用 |
| **多租户支持** | 一个网关服务多个用户 | 家庭/工作分离 |

## 📋 核心概念

### 会话键（Session Key）格式

OpenClaw 使用**层次化的会话键**来组织对话：

```
格式: agent:<agentId>:<scope>:<type>:<id>
                      │         │      │
                      │         │      └── 具体标识
                      │         └───────── 类型
                      └─────────────────── 范围
```

**示例会话键**：

| 类型 | 格式 | 说明 |
|------|------|------|
| **DM（主会话）** | `agent:main:main` | 所有私聊合并到主会话 |
| **DM（隔离）** | `agent:main:whatsapp:dm:+1555...` | 按渠道隔离 |
| **群组** | `agent:main:telegram:group:-1001...` | 群组独立会话 |
| **频道** | `agent:main:discord:channel:123...` | 频道独立会话 |

> **💡 专家提示**：主会话（`main`）是**单用户场景**的最佳选择——跨渠道的对话连续性。

### DM 会话作用域（`dmScope`）

控制私聊如何分组：

| 模式 | 会话键格式 | 适用场景 |
|------|-----------|----------|
| **`main`** | `agent:<agentId>:<mainKey>` | 单用户、跨渠道连续性 |
| **`per-peer`** | `agent:<agentId>:dm:<peerId>` | 多用户、跨渠道 |
| **`per-channel-peer`** | `agent:<agentId>:<channel>:dm:<peerId>` | 多用户收件箱（推荐） |
| **`per-account-channel-peer`** | `agent:<agentId>:<channel>:<accountId>:dm:<peerId>` | 多账户收件箱（推荐） |

**可视化对比**：

```
┌─────────────────────────────────────────────────────────────────┐
│                    dmScope 模式对比                             │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  模式 A: main（单用户）                                        │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                                                         │   │
│  │  Alice (WhatsApp) ─┐                                     │   │
│  │  Alice (Telegram) ─┼──► 共享同一个会话                    │   │
│  │  Alice (Discord)  ─┘     (agent:main:main)               │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  模式 B: per-channel-peer（多用户隔离）                       │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                                                         │   │
│  │  Alice (WhatsApp) ──────────────────────────────────►   │   │
│  │     (agent:main:whatsapp:dm:alice)                       │   │
│  │                                                         │   │
│  │  Bob (WhatsApp) ─────────────────────────────────────►   │   │
│  │     (agent:main:whatsapp:dm:bob)                         │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 🔐 安全 DM 模式（多用户场景必读）

### 为什么需要安全 DM 模式？

**问题场景**：

> Alice 发消息给 AI："[隐私] 我的医疗预约号码是 123-456-7890"
>
> Bob 随后问 AI："我们刚才在讨论什么？"
>
> 如果使用默认 `dmScope: "main"`，AI 可能会向 Bob 泄露 Alice 的私人信息！

### 配置安全 DM 模式

```json5
// ~/.openclaw/openclaw.json
{
  session: {
    // 安全 DM 模式：按渠道 + 发送者隔离
    dmScope: "per-channel-peer",
  },
}
```

### 何时启用安全 DM 模式？

| 场景 | 是否需要 | 推荐配置 |
|------|----------|----------|
| **个人使用** | ❌ 不需要 | `dmScope: "main"` |
| **单收件箱，单配对** | ❌ 不需要 | `dmScope: "main"` |
| **多配对** | ✅ **需要** | `dmScope: "per-channel-peer"` |
| **开放 DM 策略** | ✅ **需要** | `dmScope: "per-channel-peer"` |
| **多账户收件箱** | ✅ **需要** | `dmScope: "per-account-channel-peer"` |

> **🚨 安全警告**：如果你的代理可以接收**多个人的私信**，强烈建议启用安全 DM 模式！

### 配置验证

```bash
# 检查当前 DM 安全设置
openclaw security audit

# 检查配对状态
openclaw pairing list --all
```

## 💾 状态存储位置

### 文件结构

```
~/.openclaw/
├── agents/
│   └── <agentId>/
│       └── sessions/
│           ├── sessions.json              # 会话索引（键 → 元数据）
│           ├── agent:main:main.jsonl      # 主会话对话记录
│           ├── agent:main:whatsapp:...jsonl
│           └── agent:main:telegram:...jsonl
```

### 存储文件说明

| 文件/目录 | 作用 | 安全敏感度 |
|----------|------|------------|
| `sessions.json` | 会话索引（键 → 元数据） | 低 |
| `*.jsonl` | 完整对话记录 | 🔴 **高** |
| `*.deleted.*` | 已删除会话的归档 | 中 |

> **🔐 安全提醒**：`sessions/` 目录包含完整对话历史，请确保目录权限仅对您可见：
> ```bash
> chmod -R 700 ~/.openclaw/agents/*/sessions
> ```

## 🔄 会话生命周期

### 生命周期阶段

```
┌─────────────────────────────────────────────────────────────────┐
│                    会话生命周期                                  │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  1. 创建                                                        │
│     └── 首次收到消息时创建会话                                  │
│                                                                 │
│  2. 活跃                                                        │
│     └── 每条消息更新会话时间戳                                  │
│     └── AI 回复时添加 Token 计数                                │
│                                                                 │
│  3. 重置                                                        │
│     ├── 每日重置：凌晨 4:00（默认）                            │
│     ├── 空闲重置：超过配置的空闲时间                          │
│     └── 手动重置：/new 或 /reset 命令                          │
│                                                                 │
│  4. 清理                                                        │
│     └── 会话维护策略自动清理过期会话                            │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 重置策略配置

```json5
{
  session: {
    reset: {
      mode: "daily",           // daily | idle
      atHour: 4,                // 每日重置时间（小时）
      idleMinutes: 120,         // 空闲重置（分钟）
    },
    // 按类型覆盖
    resetByType: {
      direct: { mode: "idle", idleMinutes: 240 },
      group: { mode: "daily", atHour: 4 },
      thread: { mode: "daily", atHour: 4 },
    },
    // 按渠道覆盖
    resetByChannel: {
      discord: { mode: "idle", idleMinutes: 10080 },
    },
    // 重置触发器
    resetTriggers: ["/new", "/reset"],
  },
}
```

### 重置模式对比

| 模式 | 说明 | 适用场景 |
|------|------|----------|
| **`daily`** | 每天固定时间重置 | 常规对话 |
| **`idle`** | 无活动 N 分钟后重置 | 长时间运行的会话 |
| **组合** | 两者先到为准 | 精细控制 |

## 🛠️ 会话维护

### 维护策略

OpenClaw 自动应用维护策略以保持存储有界：

| 配置项 | 默认值 | 说明 |
|--------|--------|------|
| `mode` | `warn` | `warn`（报告）或 `enforce`（执行） |
| `pruneAfter` | `30d` | 删除超过 N 天的会话 |
| `maxEntries` | `500` | 最大会话数量 |
| `rotateBytes` | `10mb` | 旋转 `sessions.json` 的大小阈值 |

### 推荐配置（生产环境）

```json5
{
  session: {
    maintenance: {
      mode: "enforce",              // 生产环境必须用 enforce
      pruneAfter: "45d",             // 保留 45 天内的会话
      maxEntries: 800,               // 最多 800 个会话
      rotateBytes: "20mb",           // 20MB 后旋转索引
      resetArchiveRetention: "14d",  // 归档保留 14 天
    },
  },
}
```

### 磁盘预算配置

```json5
{
  session: {
    maintenance: {
      // 启用磁盘预算
      maxDiskBytes: "1gb",           // 总磁盘限额 1GB
      highWaterBytes: "800mb",       // 80% 触发清理
    },
  },
}
```

### 维护命令

```bash
# 预览将要清理的内容（不执行）
openclaw sessions cleanup --dry-run

# 执行清理
openclaw sessions cleanup --enforce

# 查看详细输出
openclaw sessions cleanup --dry-run --json
```

## 🐛 故障排查

### 常见问题

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| 会话混合 | `dmScope: "main"` 在多用户环境 | 改为 `per-channel-peer` |
| Token 溢出 | 历史消息过多 | 运行 `/compact` 或调整 `historyLimit` |
| 上下文丢失 | 会话重置太频繁 | 增加 `idleMinutes` |
| 存储过大 | 旧会话未清理 | 启用 `maintenance.mode: "enforce"` |
| 隐私泄露 | 多人共用主会话 | 配置安全 DM 模式 |
