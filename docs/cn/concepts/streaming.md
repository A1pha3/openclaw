## 💡 为什么需要流式传输？

### 类比：流式传输就像"实时打字"

```
┌─────────────────────────────────────────────────────────────────────────┐
│                    消息传输类比                                          │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  传统传输（一次性发送）：                                                │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │  用户发问 ..................................................      │   │
│  │  （等待 10 秒）                                                  │   │
│  │  AI：这是完整回复！（一大段文字一次性出现）                      │   │
│  │                                                                 │   │
│  │  问题：                                                          │   │
│  │  • 用户不知道 AI 是否在工作                                     │   │
│  │  • 长等待体验不好                                               │   │
│  │  • 无法提前看到内容                                             │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
│  流式传输（实时打字效果）：                                              │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │  用户发问 ..................................................      │   │
│  │  AI：好的，让我来...                                            │   │
│  │  AI：解释一下这个问题...                                       │   │
│  │  AI：首先我们需要...                                           │   │
│  │  AI：[继续发送更多内容]                                        │   │
│  │                                                                 │   │
│  │  优势：                                                          │   │
│  │  ✅ 用户立即看到响应                                            │   │
│  │  ✅ 更自然的交互体验                                            │   │
│  │  ✅ 可以提前阅读部分内容                                        │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### 两层流式传输

OpenClaw 有两个独立的流式传输层：

| 层级 | 类型 | 说明 | 支持渠道 |
|------|------|------|----------|
| **分块流式传输** | 渠道消息 | 发送完成的消息块 | 所有渠道 |
| **草稿流式传输** | 类令牌 | 实时更新草稿气泡 | Telegram 仅限 |

> **注意**：目前没有真正的令牌级流式传输到外部渠道。

## 📦 分块流式传输

### 工作原理

```
┌─────────────────────────────────────────────────────────────────────────┐
│                    分块流式传输流程                                     │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  模型生成输出                                                           │
│     │                                                                   │
│     ▼                                                                   │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │  text_delta/events（增量事件）                                  │   │
│  │  • 模型生成的文本增量                                          │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│     │                                                                   │
│     ▼                                                                   │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │  分块器（Chunker）                                               │   │
│  │  • 应用 minChars/maxChars 边界                                  │   │
│  │  • 根据断点偏好分割                                             │   │
│  │  • 维护代码围栏完整性                                            │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│     │                                                                   │
│     ▼                                                                   │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │  分块时机                                                        │   │
│  │  • text_end：立即发送块                                         │   │
│  │  • message_end：消息结束时发送所有缓冲块                        │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│     │                                                                   │
│     ▼                                                                   │
│  渠道发送（块回复）                                                      │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### 控制参数

| 参数 | 位置 | 默认值 | 说明 |
|------|------|--------|------|
| `blockStreamingDefault` | `agents.defaults` | `"off"` | 全局启用/禁用 |
| `blockStreaming` | `channels.*` | - | 渠道级别覆盖 |
| `blockStreamingBreak` | `agents.defaults` | `"message_end"` | 分块时机 |
| `blockStreamingChunk` | `agents.defaults` | `{minChars, maxChars}` | 分块大小 |
| `blockStreamingCoalesce` | `agents.defaults` | `{minChars, maxChars, idleMs}` | 合并设置 |
| `textChunkLimit` | `channels.*` | 渠道相关 | 硬性上限 |

## 🧱 分块算法

### 边界语义

```
┌─────────────────────────────────────────────────────────────────────────┐
│                    分块边界对比                                         │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  text_end（细粒度）：                                                    │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │  [模型输出] 这是一段文 ──► 发送块 1                              │   │
│  │  [模型输出] 本，这是第二段 ──► 发送块 2                           │   │
│  │  [模型输出] ，这是第三段。                                        │   │
│  │                                                                 │   │
│  │  特点：更频繁的更新，用户更快看到内容                            │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
│  message_end（粗粒度）：                                                 │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │  [模型输出] 这是一段文本，这是第二段，这是第三段。               │   │
│  │  （等待模型完成）                                                 │   │
│  │  ──► 发送完整消息（如果太长可能分多个块）                        │   │
│  │                                                                 │   │
│  │  特点：更少的消息，更整洁的体验                                  │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### 低/高边界

| 边界 | 行为 |
|------|------|
| **低边界（minChars）** | 缓冲区达到此大小前不发出 |
| **高边界（maxChars）** | 优先在此大小前分割，强制时在此处分割 |

### 断点偏好

```
断点优先级（从高到低）：

paragraph → newline → sentence → whitespace → 硬断点

示例：
"这是第一段。\n\n这是第二段的第一句。这是第二句。"

断点选择：
1. paragraph: "\n\n"（段落边界）
2. newline: "\n"（换行）
3. sentence: "。"（句子结束）
4. whitespace: " "（空格）
5. 硬断点：强制在字符处分割
```

### 代码围栏保护

```
 fenced code block
 ┌─────────────────────────────────────────────────────────────────┐
 │  ```python                                                      │
 │  def hello():                                                  │
 │      print("Hello, World!")                                    │
 │  ```                                                           │
 │  ← 代码块内不会分割，保持 Markdown 有效性                        │
 └─────────────────────────────────────────────────────────────────┘
```

## 🔄 合并策略

### 为什么需要合并？

```
┌─────────────────────────────────────────────────────────────────────────┐
│                    合并前后对比                                         │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  不合并（单行刷屏）：                                                    │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │  AI: 好的                                                      │   │
│  │  AI: 让我                                                     │   │
│  │  AI: 解释                                                     │   │
│  │  AI: 一下                                                     │   │
│  │  AI: 这个问题                                                 │   │
│  │                                                                 │   │
│  │  问题：太多短消息，刷屏                                         │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
│  合并后（渐进式输出）：                                                  │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │  AI: 好的，让我解释一下这个问题...                             │   │
│  │  AI: 首先，我们需要...                                         │   │
│  │  AI: 接下来，...                                               │   │
│  │                                                                 │   │
│  │  优势：更自然，减少消息数量                                     │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### 合并参数

```json5
{
  agents: {
    defaults: {
      blockStreamingCoalesce: {
        minChars: 500,      // 最小累积字符数
        maxChars: 2000,     // 最大缓冲字符数
        idleMs: 300         // 空闲后发送（毫秒）
      }
    }
  }
}
```

| 参数 | 说明 |
|------|------|
| `minChars` | 防止微小片段，累积足够文本再发送 |
| `maxChars` | 缓冲区上限，超过时立即刷新 |
| `idleMs` | 空闲超时，超时后发送缓冲内容 |

## ⏱️ 类人节奏延迟

### 工作原理

```
┌─────────────────────────────────────────────────────────────────────────┐
│                    节奏延迟效果                                         │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  无延迟（机器感）：                                                      │
│  AI: [第一块] → [立即] → [第二块] → [立即] → [第三块]                   │
│                                                                         │
│  自然延迟（类人）：                                                      │
│  AI: [第一块] → [等待 1.2 秒] → [第二块] → [等待 1.8 秒] → [第三块]      │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### 配置模式

```json5
{
  agents: {
    defaults: {
      humanDelay: {
        mode: "natural"  // off | natural | custom
      }
    }
  }
}
```

| 模式 | 延迟范围 | 说明 |
|------|----------|------|
| `off` | 无 | 禁用延迟 |
| `natural` | 800-2500ms | 随机延迟，模拟打字 |
| `custom` | 自定义 | 使用 `minMs`/`maxMs` |

### 自定义示例

```json5
{
  agents: {
    defaults: {
      humanDelay: {
        mode: "custom",
        minMs: 500,
        maxMs: 1500
      }
    }
  }
}
```

## 📱 Telegram 草稿流式传输

### 特殊机制

```
┌─────────────────────────────────────────────────────────────────────────┐
│                    Telegram 草稿流式传输                                │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  私聊 + 主题                                                            │
│     │                                                                   │
│     ▼                                                                   │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │  sendMessageDraft（草稿气泡）                                    │   │
│  │  • 不是真正的消息                                               │   │
│  │  • 实时更新显示                                                 │   │
│  │  • 用户看到"正在输入..."效果                                    │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│     │                                                                   │
│     ▼                                                                   │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │  streamMode 模式                                                │   │
│  │  • partial：最新文本替换                                       │   │
│  │  • block：分块更新                                              │   │
│  │  • off：禁用                                                   │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│     │                                                                   │
│     ▼                                                                   │
│  最终回复（普通消息）                                                    │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### 配置示例

```json5
{
  channels: {
    telegram: {
      streamMode: "partial",  // partial | block | off
      draftChunk: {
        minChars: 200,
        maxChars: 800
      }
    }
  }
}
```

### 模式对比

| 模式 | 行为 | 适用场景 |
|------|------|----------|
| `partial` | 用最新文本更新草稿 | 最流畅的实时体验 |
| `block` | 分块更新草稿 | 更可控的分块 |
| `off` | 无草稿流式传输 | 禁用草稿功能 |

## 🎛️ 完整配置示例

### 基础分块配置

```json5
{
  agents: {
    defaults: {
      blockStreamingDefault: "on",
      blockStreamingBreak: "text_end",
      blockStreamingChunk: {
        minChars: 300,
        maxChars: 1500,
        breakPreference: "paragraph"  // paragraph | newline | sentence | whitespace
      }
    }
  }
}
```

### 启用合并

```json5
{
  agents: {
    defaults: {
      blockStreamingDefault: "on",
      blockStreamingCoalesce: {
        minChars: 500,
        maxChars: 2000,
        idleMs: 300
      }
    }
  }
}
```

### 渠道特定配置

```json5
{
  agents: {
    defaults: {
      blockStreamingDefault: "off"
    }
  },
  channels: {
    whatsapp: {
      blockStreaming: true,  // 覆盖默认值
      textChunkLimit: 4096
    },
    telegram: {
      streamMode: "partial"
    },
    discord: {
      blockStreaming: true,
      maxLinesPerMessage: 17
    }
  }
}
```

### 自然延迟配置

```json5
{
  agents: {
    defaults: {
      blockStreamingDefault: "on",
      humanDelay: {
        mode: "natural"  // 800-2500ms 随机延迟
      }
    }
  }
}
```

## 🐛 故障排查

### 常见问题

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| 没有分块消息 | `blockStreamingDefault: "off"` | 设置为 `"on"` |
| 渠道没有分块 | 渠道未启用 | 设置 `channels.*.blockStreaming: true` |
| 消息被截断 | 超过 `textChunkLimit` | 增加限制或减少块大小 |
| 单行刷屏 | 合并未配置 | 启用 `blockStreamingCoalesce` |
| Telegram 无草稿 | 不支持草稿 | 仅私聊+主题支持 |

### 调试技巧

```bash
# 启用详细日志
openclaw gateway --verbose

# 查看流式事件
# 日志会显示 chunk 发送时机
```

## 🎯 最佳实践

### 配置建议

| 场景 | 推荐配置 | 原因 |
|------|----------|------|
| 一般使用 | `blockStreamingDefault: "off"` | 更整洁的消息历史 |
| 长回复 | `blockStreamingDefault: "on"` + `message_end` | 用户知道 AI 在工作 |
| 实时协作 | `blockStreamingDefault: "on"` + `text_end` | 最快反馈 |
| Telegram | `streamMode: "partial"` | 最佳用户体验 |

### 分块大小建议

```
┌─────────────────────────────────────────────────────────────────────────┐
│                    分块大小选择                                         │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  小块（300-500 字符）                                                   │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │  • 快速反馈                                                      │   │
│  │  • 更多消息数量                                                 │   │
│  │  • 适合：实时协作、快速问答                                     │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
│  中块（800-1500 字符）                                                  │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │  • 平衡反馈和消息数量                                           │   │
│  │  • 适合：大多数场景                                             │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
│  大块（2000+ 字符）                                                     │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │  • 较少消息数量                                                 │   │
│  │  • 更慢的首次反馈                                               │   │
│  │  • 适合：长篇内容、讲座风格                                     │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### 渠道特性

| 渠道 | 建议 | 注意 |
|------|------|------|
| **WhatsApp** | 使用合并 | 避免消息刷屏 |
| **Telegram** | 使用草稿流式传输 | 最佳实时体验 |
| **Discord** | 注意行数限制 | 默认 17 行/消息 |
| **Slack** | 启用合并 | 工作场景整洁 |
