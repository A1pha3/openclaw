## 💡 为什么需要令牌汇聚点？

### 类比：令牌汇聚就像"钥匙管理"

```
┌─────────────────────────────────────────────────────────────────────────┐
│                    令牌汇聚类比                                         │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  没有汇聚点的问题：                                                      │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │  Claude Code ──► 获取令牌 A                                      │   │
│  │  OpenClaw ────► 获取令牌 B                                      │   │
│  │                                                                 │   │
│  │  问题：                                                          │   │
│  │  • 新令牌使旧令牌失效                                          │   │
│  │  • 一个应用"登出"后另一个也失效                               │   │
│  │  • 无法预测哪个令牌有效                                       │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
│  有汇聚点的方案：                                                        │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │  令牌汇聚点（auth-profiles.json）                                │   │
│  │  ├── Claude Code 读取/写入                                       │   │
│  │  ├── OpenClaw 读取/写入                                         │   │
│  │  ├── Codex CLI 读取/写入                                       │   │
│  │                                                                 │   │
│  │  优势：                                                          │   │
│  │  ✅ 统一令牌存储                                                │   │
│  │  ✅ 避免令牌冲突                                                │   │
│  │  ✅ 确定性路由                                                  │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### 核心价值

| 价值 | 说明 |
|------|------|
| **令牌同步** | 多个工具共享同一令牌源 |
| **避免冲突** | 减少令牌失效导致的"登出"问题 |
| **灵活路由** | 支持多配置文件和账户隔离 |

## 📁 令牌存储位置

### 存储结构

```
┌─────────────────────────────────────────────────────────────────────────┐
│                    令牌存储结构                                         │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  ~/.openclaw/                                                         │
│  ├── agents/                                                          │
│  │   └── <agentId>/                                                   │
│  │       ├── agent/                                                    │   │
│  │       │   ├── auth-profiles.json ───► 认证配置文件（OAuth + API 密钥）│   │
│  │       │   └── auth.json ───────────► 运行时缓存（自动管理）        │   │
│  │       └── sessions/                                                 │   │
│  │                                                                   │   │
│  └── credentials/                                                       │   │
│      └── oauth.json ─────────────────► 旧版存储（首次导入）           │   │
│                                                                         │
│  注意：所有路径遵循 $OPENCLAW_STATE_DIR 覆盖                            │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### 文件说明

| 文件 | 类型 | 说明 |
|------|------|------|
| `auth-profiles.json` | 持久存储 | OAuth + API 密钥的汇聚点 |
| `auth.json` | 运行时缓存 | 自动管理，请勿编辑 |
| `oauth.json` | 旧版存储 | 首次使用时导入到新位置 |

## 🔐 Anthropic setup-token

### 认证流程

```
┌─────────────────────────────────────────────────────────────────────────┐
│                    setup-token 流程                                     │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  1. 生成 setup-token                                                     │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │  claude setup-token                                               │   │
│  │  │                                                             │   │
│  │  ▼                                                             │   │
│  │  生成一个长期令牌                                               │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│     │                                                                   │
│     ▼                                                                   │
│  2. 导入到 OpenClaw                                                     │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │  openclaw models auth setup-token --provider anthropic          │   │
│  │  │                                                             │   │
│  │  ▼                                                             │   │
│  │  粘贴令牌                                                        │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│     │                                                                   │
│     ▼                                                                   │
│  3. 存储为令牌认证配置文件                                               │
│     │                                                                   │
│     ▼                                                                   │
│  验证：openclaw models status                                            │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### 命令使用

```bash
# 交互式导入
openclaw models auth setup-token --provider anthropic

# 手动粘贴（如果在其他地方生成）
openclaw models auth paste-token --provider anthropic

# 验证状态
openclaw models status
```

## 🔄 OpenAI Codex OAuth（PKCE）

### 完整流程

```
┌─────────────────────────────────────────────────────────────────────────┐
│                    PKCE OAuth 流程                                       │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  1. 生成 PKCE 参数                                                      │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │  • 生成验证器（verifier）                                        │   │
│  │  • 生成质询（challenge）                                          │   │
│  │  • 生成随机 state                                               │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│     │                                                                   │
│     ▼                                                                   │
│  2. 打开授权页面                                                        │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │  https://auth.openai.com/oauth/authorize?                        │   │
│  │    ?client_id=...                                                │   │
│  │    &redirect_uri=http://127.0.0.1:1455/auth/callback                 │   │
│  │    &scope=...                                                   │   │
│  │    &response_type=code                                           │   │
│  │    &code_challenge=...                                          │   │
│  │    &code_challenge_method=S256                                   │   │
│  │    &state=...                                                   │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│     │                                                                   │
│     ▼                                                                   │
│  3. 用户授权并回调                                                      │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │  • 尝试本地回调捕获（http://127.0.0.1:1455/auth/callback）       │   │
│  │  • 失败则手动粘贴重定向 URL 或代码                              │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│     │                                                                   │
│     ▼                                                                   │
│  4. 令牌交换                                                            │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │  POST https://auth.openai.com/oauth/token                         │   │
│  │  • 授权码 + 验证器                                               │   │
│  │  • 换取 { access, refresh, expires, accountId }                    │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│     │                                                                   │
│     ▼                                                                   │
│  5. 存储凭据                                                           │
│     │                                                                   │
│     ▼                                                                   │
│  完成 ✓                                                                 │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### 向导路径

```bash
# 启动向导
openclaw onboard

# 选择认证
# → 认证选择 → openai-codex
```

## 🔄 令牌刷新与过期

### 自动刷新机制

```
┌─────────────────────────────────────────────────────────────────────────┐
│                    令牌刷新流程                                         │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  运行时检查令牌                                                         │
│     │                                                                   │
│     ▼                                                                   │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │  检查 expires 时间戳                                              │   │
│  │                                                                 │   │
│  │  expires 在未来？                                                │   │
│  │  │                                                               │   │
│  │  YES ──► 使用已存储的访问令牌                                  │   │
│  │  │                                                               │   │
│  │  NO ──► 刷新令牌（文件锁保护）                                  │   │
│  │         │                                                       │   │
│  │         ▼                                                       │   │
│  │    ┌─────────────────────────────────────────────────────┐   │   │
│  │    │  使用 refresh_token 获取新令牌                          │   │   │
│  │    │  更新 auth-profiles.json                                 │   │   │
│  │    └─────────────────────────────────────────────────────┘   │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
│  注意：刷新流程是自动的，通常不需要手动管理                            │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### 时间戳说明

| 字段 | 说明 |
|------|------|
| `expires` | 令牌过期时间戳 |
| 自动检测 | 运行时自动检查并刷新 |
| 文件锁 | 刷新时使用锁避免并发冲突 |

## 👥 多账户模式

### 模式对比

```
┌─────────────────────────────────────────────────────────────────────────┐
│                    多账户模式对比                                       │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  模式 1：独立智能体（推荐）                                           │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │  创建隔离的智能体：                                              │   │
│  │  • work ──► 工作账户 + 工作会话                                │   │
│  │  • personal ──► 个人账户 + 个人会话                              │   │
│  │                                                                 │   │
│  │  优势：                                                          │   │
│  │  ✅ 完全隔离，永不相交                                          │   │
│  │  ✅ 独立的会话和凭据                                            │   │
│  │  ✅ 简单清晰                                                    │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
│  模式 2：单智能体多配置文件                                             │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │  同一智能体，多个配置文件：                                     │   │
│  │  • anthropic:work ──► 工作账户                                 │   │
│  │  • anthropic:personal ──► 个人账户                             │   │
│  │                                                                 │   │
│  │  选择方式：                                                      │   │
│  │  • auth.order 配置默认顺序                                     │   │
│  │  • /model ...@<profileId> 会话覆盖                              │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### 独立智能体配置

```bash
# 创建隔离的智能体
openclaw agents add work
openclaw agents add personal

# 为每个智能体配置认证
# （通过向导或手动配置）

# 路由聊天到正确的智能体
# （通过 binding 配置）
```

### 多配置文件路由

```json5
{
  // 全局配置文件顺序
  auth: {
    order: ["anthropic:work", "anthropic:personal"]
  },
  agents: {
    defaults: {
      model: "anthropic/claude-sonnet-4-20250514@anthropic:work"
    }
  }
}
```

### 会话覆盖

```
# 使用特定配置文件
/model Opus@anthropic:work
/model Sonnet@anthropic:personal

# 查看可用配置文件
openclaw channels list --json
```

## 🐛 故障排查

### 常见问题

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| 随机"登出" | 多个工具分别获取令牌 | 统一使用 `auth-profiles.json` |
| 刷新失败 | 令牌已撤销 | 重新认证 |
| 配置冲突 | 多配置文件路由错误 | 检查 `auth.order` |
| 令牌过期 | 超过有效期 | 自动刷新或重新认证 |

### 调试技巧

```bash
# 查看认证状态
openclaw models status --verbose

# 检查配置文件
openclaw config get auth

# 查看令牌过期时间
openclaw models status --json | jq '.auth.oauth'
```

## 🎯 最佳实践

### 认证建议

| 实践 | 说明 |
|------|------|
| **使用独立智能体** | 个人和工作完全隔离 |
| **统一令牌源** | 避免多工具冲突 |
| **定期验证** | 使用 `models status --check` |
| **安全存储** | 使用 `auth-profiles.json` 统一管理 |

