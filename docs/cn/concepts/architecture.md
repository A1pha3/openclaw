### 设计决策背景

```
┌─────────────────────────────────────────────────────────────────┐
│                    架构设计决策                                  │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  问题：如何设计一个统一的 AI 代理网关系统？                      │
│                                                                 │
│  可选方案：                                                      │
│  1. 每个渠道独立运行 → 简单但无法统一管理                        │
│  2. 多个 Gateway 实例 → 复杂且资源浪费                          │
│  3. 单一 Gateway 网关（✓ 推荐方案）                             │
│                                                                 │
│  最终选择：方案 3                                               │
│  理由：                                                          │
│  ✅ 统一控制平面                                                │
│  ✅ 高效资源利用                                                │
│  ✅ 简化部署和管理                                              │
│  ✅ 安全的设备配对机制                                          │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 核心设计原则

| 原则 | 说明 | 好处 |
|------|------|------|
| **单一 Gateway** | 每台主机一个网关 | 避免 WhatsApp 会话冲突 |
| **WebSocket 优先** | 所有客户端通过 WS 连接 | 实时双向通信 |
| **类型化协议** | JSON Schema 验证所有消息 | 类型安全和可预测性 |
| **设备配对** | 新设备需要批准 | 安全的多设备访问 |

### 组件详解

#### 1. Gateway 网关（守护进程）

**职责**：

| 功能 | 说明 |
|------|------|
| **维护提供商连接** | 管理 WhatsApp/Telegram 等渠道连接 |
| **WebSocket API** | 暴露类型化的 WS 接口 |
| **消息验证** | 根据 JSON Schema 验证入站帧 |
| **事件分发** | 发出 agent/chat/presence/health/heartbeat/cron 事件 |

#### 2. 控制平面客户端

**类型**：

| 客户端 | 用途 |
|--------|------|
| **macOS 应用** | 图形界面控制 |
| **CLI** | 命令行管理 |
| **Web 管理界面** | 浏览器控制 |

**特性**：
- 每个客户端一个 WS 连接
- 发送请求：`health`、`status`、`send`、`agent`、`system-presence`
- 订阅事件：`tick`、`agent`、`presence`、`shutdown`

#### 3. 节点（Node）

**角色定义**：

```javascript
// 节点连接时声明身份
{
  role: "node",
  capabilities: ["canvas", "camera", "screen", "location"],
  commands: ["canvas.*", "camera.*", "screen.record", "location.get"]
}
```

**特性**：
- 以 `role: node` 连接到同一个 WS 服务器
- 基于设备的配对机制
- 暴露设备专用命令

## 🔄 连接生命周期

### 单个客户端连接流程

```
Client                    Gateway
  │                          │
  │---- req:connect -------->│  1. 发送连接请求
  │                          │     （包含设备身份）
  │<------ res (ok) ---------│  2. 接收响应
  │   payload=hello-ok       │     （携带快照）
  │   (presence + health)    │
  │                          │
  │<------ event:presence ---│  3. 接收在线事件
  │<------ event:tick -------│  4. 接收心跳事件
  │                          │
  │------- req:agent ------->│  5. 发送 AI 请求
  │<------ res:agent --------│  6. 接收确认
  │   ack: {runId,           │     {runId, status: "accepted"}
  │         status:"accepted"}│
  │<------ event:agent ------│  7. 接收流式响应
  │   (streaming)            │
  │<------ res:agent --------│  8. 接收最终结果
  │   {runId, status,         │     {runId, status, summary}
  │    summary}               │
```

## 🔌 线路协议（Wire Protocol）

### 协议摘要

| 特性 | 说明 |
|------|------|
| **传输层** | WebSocket（文本帧） |
| **数据格式** | JSON 载荷 |
| **首帧要求** | **必须**是 `connect` |
| **认证** | `connect.params.auth.token` 必须匹配（如设置了令牌） |

### 请求格式

```json5
// 请求消息
{
  type: "req",
  id: "unique-request-id",
  method: "agent",
  params: { /* 方法参数 */ }
}
```

### 响应格式

```json5
// 响应消息
{
  type: "res",
  id: "unique-request-id",
  ok: true,
  payload: { /* 响应数据 */ }
}
```

### 事件格式

```json5
// 服务器推送事件
{
  type: "event",
  event: "agent",        // 事件名称
  payload: { /* 事件数据 */ },
  seq: 1,                // 序列号（可选）
  stateVersion: 1        // 状态版本（可选）
}
```

### 事件类型

| 事件 | 触发时机 | 载荷内容 |
|------|----------|----------|
| **agent** | AI 响应增量 | 流式文本块 |
| **chat** | 收到聊天消息 | 消息内容 |
| **presence** | 设备上线/下线 | 在线状态 |
| **health** | 健康状态变化 | 检查结果 |
| **heartbeat** | 定期心跳 | 时间戳 |
| **cron** | 定时任务触发 | 任务信息 |
| **shutdown** | Gateway 关闭 | 关闭原因 |
| **tick** | 定期系统状态 | 状态快照 |

### 幂等键（去重）

有副作用的方法（`send`、`agent`）需要幂等键以安全重试：

```json5
{
  type: "req",
  id: "req-id",
  method: "send",
  params: {
    idempotencyKey: "unique-key-for-retry",  // 幂等键
    // ... 其他参数
  }
}
```

### 信任级别

| 连接类型 | 验证要求 | 自动批准 |
|----------|----------|----------|
| **本地（loopback）** | 设备 ID | ✅ 是 |
| **本地（tailnet）** | 设备 ID | ✅ 是 |
| **远程** | 签名 + 明确批准 | ❌ 否 |

## 🐛 故障排查

### 常见问题

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| 连接被拒绝 | `connect` 不是首帧 | 确保第一个消息是 `connect` |
| 认证失败 | Token 不匹配 | 检查 `gateway.auth.token` 配置 |
| 设备配对超时 | 远程连接未批准 | 通过 CLI 批准配对请求 |
| WebSocket 断开 | 网络问题或心跳超时 | 检查网络连接，增加心跳间隔 |

### 调试技巧

```bash
# 启用详细日志
openclaw gateway --verbose
```

## 🚫 不变量（Invariants）

| 不变量 | 说明 | 违反后果 |
|--------|------|----------|
| **单一 Gateway** | 每台主机恰好一个 Gateway | WhatsApp 会话冲突 |

| **设备认证** | 所有连接需要设备身份 | 安全风险 |

## 🏆 架构设计原则

### 核心原则

1. **单一职责**: Gateway 专注于连接管理和消息路由
2. **无状态设计**: 状态存储在外部，便于扩展
3. **最终一致性**: 接受短暂不一致，保证最终一致
4. **容错设计**: 假设故障会发生，优雅降级
5. **可观测性**: 所有操作可追踪、可监控

### 反模式（避免）

| 反模式 | 问题 | 正确做法 |
|--------|------|----------|
| 多 Gateway 共享会话 | 消息重复/丢失 | 单一 Gateway 原则 |
| 同步阻塞调用 | 性能瓶颈 | 异步事件驱动 |
| 硬编码配置 | 难以维护 | 配置外部化 |
| 忽略监控 | 故障难定位 | 全面可观测性 |
