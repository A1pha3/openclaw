## 💡 为什么需要重试策略？

### 类比：重试就像"重新拨打"

```
┌─────────────────────────────────────────────────────────────────────────┐
│                    重试策略类比                                         │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  现实场景：打电话                                                        │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │  拨打电话 ──► 忙音 ──► 挂断 ──► 再拨一次 ──► 接通 ✓              │   │
│  │                                                                 │   │
│  │  好的做法：                                                       │   │
│  │  • 等待一会儿再拨（避免立即重拨）                                │   │
│  │  • 不要无休止地重拨（设置重试次数）                              │   │
│  │  • 随机等待时间（避免所有人同时重拨）                            │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
│  OpenClaw 渠道消息发送：                                                │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │  发送消息 ──► 失败（429 速率限制） ──► 等待 ──► 重试 ──► 成功 ✓   │   │
│  │                                                                 │   │
│  │  重试策略：                                                       │   │
│  │  • 指数退避（逐渐增加等待时间）                                  │   │
│  │  • 抖动（随机延迟）                                             │   │
│  │  • 最大重试次数                                                 │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### 核心原则

| 原则 | 说明 |
|------|------|
| **按请求重试** | 仅重试失败的 HTTP 请求，不重试整个流程 |
| **保持顺序** | 不跳过已完成的步骤 |
| **避免重复** | 不重复执行非幂等操作 |

## 🔄 重试机制

### 重试流程

```
┌─────────────────────────────────────────────────────────────────────────┐
│                    重试决策流程                                         │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  发送请求                                                                │
│     │                                                                   │
│     ▼                                                                   │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │  请求成功？                                                       │   │
│  │                                                                 │   │
│  │  YES ──► 完成 ✓                                                  │   │
│  │                                                                 │   │
│  │  NO ──► 检查错误类型                                            │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│     │                                                                   │
│     ▼                                                                   │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │  可重试的错误？                                                   │   │
│  │  • 429 速率限制                                                  │   │
│  │  • 超时                                                          │   │
│  │  • 连接/重置/关闭                                               │   │
│  │                                                                 │   │
│  │  NO ──► 失败 ✗                                                  │   │
│  │                                                                 │   │
│  │  YES ──► 达到重试次数上限？                                     │   │
│  │          │                                                       │   │
│  │          YES ──► 失败 ✗                                        │   │
│  │          │                                                       │   │
│  │          NO ──► 计算延迟时间                                     │   │
│  │                  │                                               │   │
│  │                  ▼                                               │   │
│  │          ┌─────────────────────────────────────────────────┐   │   │
│  │          │  指数退避 + 抖动                                    │   │   │
│  │          │  基础延迟 × 2^重试次数 × (1 ± 抖动)               │   │   │
│  │          └─────────────────────────────────────────────────┘   │   │
│  │                  │                                               │   │
│  │                  ▼                                               │   │
│  │          等待延迟 ──► 重试请求                                 │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### 延迟计算

```
延迟 = min(基础延迟 × 2^(重试次数), 最大延迟) × (1 ± 抖动)

示例（基础延迟 400ms，抖动 0.1）：
• 第 1 次重试：400ms × 2^0 = 400ms ± 40ms
• 第 2 次重试：400ms × 2^1 = 800ms ± 80ms
• 第 3 次重试：400ms × 2^2 = 1600ms ± 160ms
```

## ⚙️ 默认配置

### 全局默认值

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `attempts` | `3` | 最大重试次数 |
| `maxDelayMs` | `30000` | 最大延迟上限（30 秒） |
| `jitter` | `0.1` | 抖动系数（10%） |

### 渠道默认值

| 渠道 | `minDelayMs` | 特殊行为 |
|------|--------------|----------|
| **Telegram** | `400` | 支持 retry_after |
| **Discord** | `500` | 仅 429 错误重试 |
| **Slack** | `500` | 标准重试 |
| **WhatsApp** | `1000` | 标准重试 |

## 📡 渠道行为

### Discord

```json5
{
  channels: {
    discord: {
      retry: {
        attempts: 3,
        minDelayMs: 500,
        maxDelayMs: 30000,
        jitter: 0.1
      }
    }
  }
}
```

**重试条件**：
- ✅ HTTP 429（速率限制）
- ❌ 其他错误不重试

**特殊行为**：
- 优先使用 Discord 的 `retry_after` 头
- 回退到指数退避

### Telegram

```json5
{
  channels: {
    telegram: {
      retry: {
        attempts: 3,
        minDelayMs: 400,
        maxDelayMs: 30000,
        jitter: 0.1
      }
    }
  }
}
```

**重试条件**：
- ✅ HTTP 429（速率限制）
- ✅ 超时
- ✅ 连接/重置/关闭错误
- ✅ 暂时不可用

**特殊行为**：
- 优先使用 `retry_after`
- Markdown 解析错误不重试，回退为纯文本

## 🔧 配置示例

### 基础配置

```json5
{
  channels: {
    telegram: {
      retry: {
        attempts: 3,
        minDelayMs: 400,
        maxDelayMs: 30000,
        jitter: 0.1
      }
    }
  }
}
```

### 激进重试（快速失败）

```json5
{
  channels: {
    discord: {
      retry: {
        attempts: 2,          // 减少重试次数
        minDelayMs: 200,      // 减少初始延迟
        maxDelayMs: 5000,     // 降低最大延迟
        jitter: 0.05          // 减少抖动
      }
    }
  }
}
```

### 保守重试（高可靠性）

```json5
{
  channels: {
    telegram: {
      retry: {
        attempts: 5,          // 增加重试次数
        minDelayMs: 500,      // 增加初始延迟
        maxDelayMs: 60000,    // 提高最大延迟
        jitter: 0.2           // 增加抖动
      }
    }
  }
}
```

### 全局默认覆盖

```json5
{
  channels: {
    "*": {
      retry: {
        attempts: 3,
        minDelayMs: 500,
        maxDelayMs: 30000,
        jitter: 0.1
      }
    }
  }
}
```

## 📋 参数说明

### attempts（重试次数）

| 值 | 效果 | 适用场景 |
|------|------|----------|
| `0` | 不重试 | 快速失败场景 |
| `1-2` | 低重试 | 可靠连接 |
| `3` | 标准（默认） | 大多数场景 |
| `5+` | 高重试 | 不稳定网络 |

### minDelayMs（最小延迟）

| 值 | 效果 | 适用场景 |
|------|------|----------|
| `100-200` | 快速重试 | 本地开发 |
| `400-500` | 标准（默认） | 生产环境 |
| `1000+` | 慢速重试 | 严格速率限制 |

### maxDelayMs（最大延迟）

| 值 | 效果 |
|------|------|
| `5000` | 快速恢复 |
| `30000` | 标准（默认） |
| `60000+` | 长等待恢复 |

### jitter（抖动系数）

| 值 | 效果 |
|------|------|
| `0` | 无抖动（不推荐） |
| `0.1` | 标准（10%） |
| `0.2` | 高抖动（20%） |

## ⚠️ 注意事项

### 非幂等操作

```
┌─────────────────────────────────────────────────────────────────────────┐
│                    幂等性说明                                           │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  幂等操作（可安全重试）：                                                │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │  • 发送消息（相同内容）                                           │   │
│  │  • 上传文件                                                       │   │
│  │  • 查询状态                                                       │   │
│  │                                                                 │   │
│  │  特点：多次执行产生相同结果                                     │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
│  非幂等操作（需谨慎重试）：                                              │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │  • 添加反应表情                                                   │   │
│  │  • 发送投票                                                       │   │
│  │  • 删除消息                                                       │   │
│  │                                                                 │   │
│  │  特点：多次执行可能产生不同结果                                 │   │
│  │  注意：重试仅针对失败的 HTTP 请求                                │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### 组合流程

```
重要：重试按请求应用，不会重试已完成的步骤

示例流程：
1. 上传图片 ──► 成功 ✓
2. 发送带图片消息 ──► 失败（429）
3. 重试步骤 2 ──► 成功 ✓

不会重做步骤 1（上传图片）
```

## 🐛 故障排查

### 常见问题

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| 频繁 429 错误 | 请求过快 | 增加 `minDelayMs` |
| 消息丢失 | 重试次数不足 | 增加 `attempts` |
| 重试风暴 | 无抖动 | 启用 `jitter` |

### 调试技巧

```bash
# 启用详细日志
openclaw gateway --verbose

# 查看重试日志
grep "retry" /tmp/openclaw/openclaw-*.log

# 检查配置
openclaw config get channels.telegram.retry
```

### 错误代码处理

| 错误代码 | 含义 | 推荐配置 |
|----------|------|----------|
| `429` | 速率限制 | 增加 `minDelayMs` |
| `502` | 网关错误 | 增加重试次数 |
| `503` | 服务不可用 | 增加重试次数 |
| `504` | 超时 | 增加 `maxDelayMs` |
